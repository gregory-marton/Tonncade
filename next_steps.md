# Next Steps: Future Feature Ideas

New-capability ideas that aren't blocking the mobile experience itself — as opposed to
`mobile_redesign_plan.md`'s backlog, which is scoped to what's needed for a good, consistent
mobile experience. Numbered to match the task tracker. None of this has been designed or
planned yet.

- **#24 Sandbox mode tap-and-hold note highlighting**: tap-and-hold a cell to highlight every other cell with the same note name across the lattice, with each highlighted cell's octave-qualified note name (e.g. "C3") and frequency (e.g. "440Hz") overlaid.
- **#27 Melody mode MIDI source, post-playback transform, bundled songs**:
  - Replace the raw upload picker with a local MIDI folder the player sets once, browsed via a dropdown of its contents — no search/download feature.
  - After playing the melody, its notes stay highlighted and become a moveable shape as a whole — draggable to translate/transpose, two-finger rotate for harmonic variation — with a new control to save the altered MIDI afterward.
  - Ship a dozen or so public-domain children's songs alongside "Hot Cross Buns." Open question, explicitly for later exploration/conversation, not yet decided: bundle them as part of the app's files, or serve them from a web directory that could double as a shareable alternative to a local MIDI folder — writable web directories may not be a realistic option, so this needs more thought before designing.
- **#28 Device-rotation as a viewing angle, not just a resize** (farther future): in Gravity mode, rotating the device correctly keeps "higher pitch = up" fixed on screen, since falling pieces need a stable notion of "down" — that stays as is. In Blast, Snake, Melody, and Sandbox, none of which have gravity-driven mechanics, device rotation could instead let the Tonnetz visually rotate along with the phone (0°/90°/180°/270°), so physically turning the device reveals a different rotated view of the same fixed lattice, while controls/stats and — critically — each hex's in-grid note-name label counter-rotate individually to stay upright and legible regardless of the overall grid's rotation. Needs a real device-orientation-angle read (`screen.orientation.angle` or equivalent) with early cross-browser verification, since this API has had inconsistent behavior on iOS Safari historically.
- **#29 i18n readiness audit**: any English-language UI text (button labels, status messages, instructions) is a future translation cost. Strong bias against adding more English text where a non-linguistic alternative (icon, symbol, color) would work just as well. Some English is likely unavoidable — chord type names ("Major 7th", "Sus4", etc.) and note names use standard music-theory notation that doesn't have an obvious non-linguistic substitute, and inventing one isn't something to do without real music-theory expertise informing it. Scope: catalog existing English strings across the UI as a review list, and keep this bias in mind for new features going forward.
- **#39 Easy/Medium/Hard difficulty presets for Blast and Gravity**: a difficulty selector that swaps which piece set is in play — Easy uses only smaller pieces (predefined), Medium mixes smaller and the current full-size pieces, Hard is today's existing piece set unchanged. Not yet designed: where the selector lives in the UI, and which specific pieces count as "smaller" for Easy/Medium (needs a piece-by-piece review of `js/pieces.js`'s `Pieces.TYPES`).
- **#46 Melody mode: fix long-song timeout, and rethink the practice loop for long songs**:
  - **Bug report**: longer songs don't work — described as "the 2s timeout isn't since last tap, it's overall, which is not right." Investigated `js/midi.js`: the "going ahead" timeout (`userRepeatTimeoutId`, `handleUserInputNote`) does appear to reset on every correct note in the current code, so this specific mechanism needs live reproduction to confirm whether it's really the culprit. What's unambiguously a real scaling problem, found while investigating: `playTargetSequence()` re-schedules and replays the *entire* target segment from note 0 every single time `targetLength` grows by one (the "Simon says" pattern — each successful extension re-listens to the whole growing prefix from the start, timed relative to `melody[0].time`). For a long song this means later rounds require sitting through an ever-longer replay before getting to attempt the new note, which would plausibly feel like "some overall timer is against me" even though the specific 2000ms constant itself resets correctly per tap. Needs on-device reproduction to pin down exactly which mechanism the reported symptom traces to before fixing.
  - **Pedagogical redesign**: long songs need to be broken into smaller segments to practice, rather than one ever-growing whole-song sequence from the first note.
  - **Error tolerance**: past some difficulty threshold, forcing an exact-match full redo on every wrong note becomes counterproductive. Instead, tolerate a level of errors within a practice pass, track *where* in the song they happen, and surface that (e.g. "you tend to miss around here") to guide practice — rather than resetting progress on every mistake.
- **#47 Console errors when opened via `file://`**: the README explicitly promises `file://` support ("no server or build steps needed"), but opening `index.html` directly currently logs real errors, not just warnings — reported live via a browser console session. Two distinct causes, both inherent to the `file://` origin being `null`, not app bugs in the usual sense: (1) `navigator.serviceWorker.register()` throws "The URL protocol of the current origin ('null') is not supported" — `js/main.js`'s catch handler (`window.onload`) already special-cases the *Playwright test* "blocked" error message with a friendly log instead of `console.error`, but doesn't catch this file://-specific message, so it falls through to a scary-looking `console.error`; a small, clean fix (also check for this message, or check `location.protocol === 'file:'` and skip the registration attempt entirely up front). (2) `<link rel="manifest" href="manifest.json">` in `index.html` triggers a browser-level CORS-blocked fetch attempt (plus an "unsafe frame load" warning) — this happens before any app JS runs, so it's likely not preventable from app code without dropping the manifest tag (and PWA installability) entirely, which isn't worth it just to silence file:// console noise. Gameplay itself is unaffected either way. Probably worth a README clarification either way ("full gameplay works via file://; PWA install/offline support needs a real server") alongside the SW fix.
- **#48 Gravity playthrough recording: a genuine best-effort game, aimed at a real simultaneous multi-line clear**: the two real playthroughs recorded so far (via the console keystroke-logger built for the overlap-bug investigation) each only ever cleared one line at a time, sequentially — neither happened to complete two-or-more rows in the same lock. A third recording, played naturally rather than steered toward any specific target, specifically to capture a genuine simultaneous multi-line clear, would round out the story-test source material (single clear, multi clear, and eventually play-to-game-over are the three distinct scenarios worth having faithful recordings of).
- **#30 Alternate mathematical tuning system + selector** (farther future, nebulous): add a second mathematical backing for the grid's frequencies — an alternative to the current Tonnetz interval mapping (`Tonnetz.getMidi`: p:+7/q:+3 semitones for Standard tuning, p:-3/q:+4 for Gravity) — with a UI selector to switch between tuning systems. Prompted by an external conversation the user had exploring the idea (https://chatgpt.com/share/6a5ba607-d5e0-83ea-81c5-1796ca67c828); not yet read into or incorporated here, no design decisions made.
